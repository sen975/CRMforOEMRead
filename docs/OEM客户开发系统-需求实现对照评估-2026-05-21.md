# OEM客户开发系统需求实现对照评估

评估日期：2026-05-21

需求文档来源：`d:\WorkMatter\OEM客户管理系统 - 内部自研需求方案.docx`

代码范围：

- 前端：`apps/web`
- 后端：`apps/api`
- 数据模型：`apps/api/prisma/schema.prisma`

评估方式：

- 以代码静态审查为主，重点核对“前端页面/交互 -> API -> 后端服务 -> 数据落库”是否闭环。
- 已执行基础类型检查：`npm run lint -w @oem-crm/web`、`npm run lint -w @oem-crm/api`，两者均通过。
- 本文不等同于真实业务联调结果；涉及 SMTP/IMAP、AI、公开搜索、Redis、PostgreSQL、Playwright 爬虫的功能，还依赖环境变量与外部服务。

状态定义：

- `已完成`：前端入口、后端接口、服务逻辑、数据存储基本闭环，能支撑一期使用。
- `部分完成`：有页面/接口/数据结构，但覆盖不全、链路不通，或只完成了 MVP 的一部分。
- `未完成`：没有产品化前后端链路，或只有占位/脚本/表结构，没有可用功能。

- 企业资料库
- 客户基础管理
- 官网分析
- 背调报告
- OEM 评分
- AI 邮件草稿

## 二点五、基于 `design.md` 的接口二次核对结论

本次额外对照了后端设计/Swagger 文档草稿：

- 文档：`docs/design.md`
- 前端实际调用：`apps/web/src/pages/*`
- 后端真实控制器：`apps/api/src/modules/*/*.controller.ts`

二次核对后的判断口径如下：

- `前端和后端真实代码` 应作为“接口是否可用”的最终依据。
- `docs/design.md` 不是当前代码自动生成的 Swagger 快照，而更像设计文档/接口草案。
- 因此，接口问题需要区分为两类：
  - `真实前后端断链`：前端请求路径与后端控制器不一致，会直接导致页面不可用。
  - `设计文档过期/不一致`：`design.md` 与当前后端代码不一致，但前端和后端本身可能是对得上的。

二次核对结论：

- 当前确认的`真实前后端断链`主要是 1 处：
  - 跟进任务页前端使用 `/follow-ups*`
  - 后端真实接口为 `/follow-up-tasks*`
  - `design.md` 在这一点上与后端一致，说明这里是前端未跟上接口定义

- 当前确认的`design.md 过期或不一致`主要包括：
  - 文档写 `GET /users`、`POST /users`、`PATCH /users/:id`
  - 当前后端真实接口是 `/settings/users*`
  - 文档写 `GET /roles`
  - 当前后端真实接口是 `/settings/roles`
  - 文档写 `GET /audit-logs`
  - 当前后端真实接口是 `/settings/audit-logs`
  - 文档写 `POST /roles/:id/permissions`
  - 当前后端代码中未见该接口
  - 文档写 `PATCH /contacts/:id`、`DELETE /contacts/:id`
  - 当前后端代码中未见这两个联系人接口

- 当前确认的`后端真实存在、但 design.md 未完整覆盖`的接口包括：
  - `GET /email-drafts`
  - `GET /email-threads`
  - `GET /customers/:customerId/email-drafts`
  - `PATCH /blacklist-rules/:id`
  - `GET/POST /knowledge/brands`
  - `PATCH /knowledge/:entity/:id`

结论：

- 之前评估文档中关于“接口不一致”的核心判断总体成立。
- 但需要补充说明：不是“很多前端接口都错了”，而是“多数核心接口前后端对得上，主要问题是 `design.md` 与当前实现不完全同步，以及跟进任务页存在真实断链”。

## 三、当前最关键的断链/高风险问题

### 1. 跟进任务页前后端接口路径不一致

- 前端：`apps/web/src/pages/FollowUpsPage.tsx`
- 前端请求：`/follow-ups`、`/follow-ups/:id/complete`
- 后端实际：`apps/api/src/modules/follow-ups/follow-ups.controller.ts`
- 后端路由前缀：`/follow-up-tasks`

结论：

- 跟进任务独立页面当前大概率不可用。
- 客户详情页里的“跟进”标签只是展示 `customer.followUpTasks`，不依赖这个错误接口，因此只是一部分能看，独立任务管理页不能正常闭环。

补充说明：

- 二次对照 `docs/design.md` 后可确认，这一条不是设计文档误差。
- 在这个问题上，`design.md` 与后端控制器是一致的，真正不一致的是前端页面调用路径。

### 1.1 `design.md` 与当前后端实现不完全同步

表现：

- `design.md` 中部分接口仍使用更早期的资源路径命名，如 `/users`、`/roles`、`/audit-logs`
- 当前真实后端实现已经收敛到 `/settings/users`、`/settings/roles`、`/settings/audit-logs`
- 文档中还包含当前后端未实现的接口，如：
  - `POST /roles/:id/permissions`
  - `PATCH /contacts/:id`
  - `DELETE /contacts/:id`

结论：

- `design.md` 不能直接当作“当前真实后端 Swagger”使用。
- 若后续继续做接口核对，应优先以：
  - `apps/api/src/modules/*/*.controller.ts`
  - 运行中的 `/api/docs`
  为准。

### 2. 审计日志只有查询，没有写入

- 数据表：`AuditLog`
- 查询接口：`GET /settings/audit-logs`
- 代码中没有发现任何 `prisma.auditLog.create(...)` 或类似写入逻辑

结论：

- “所有操作留存日志，关键操作可追溯”这一条当前没有真正落地。

### 3. 附件/文件上传链路没有实现

- Schema 有 `FileAsset`、`EmailAttachment`
- 证书、案例、报价等模型里也预留了 `fileAssetId`
- 前端只允许录入 `fileAssetId` 文本，没有上传、预览、权限校验链路

结论：

- “附件上传权限管控”“证书/案例/报价附件管理”当前都不能算已实现。

### 4. 客户导入不是产品能力，只是一次性脚本

- 存在脚本：`apps/api/prisma/import-eva-customers.ts`
- 没有前端导入页面，也没有导入 API

结论：

- 需求中的“录入/导入目标客户”，目前只有“录入”产品化，“导入”还停留在开发脚本层。

### 5. 自动阶段流转没有完整记录到阶段历史

- 手工改阶段会调用 `/customers/:id/stage`，会写 `CustomerStageHistory`
- 但研究报告、OEM 评分、邮件发送、回复同步、报价、样品这些自动阶段更新，多数直接 `prisma.customer.update({ stage: ... })`

结论：

- 客户阶段虽然会变，但“阶段变更历史”并不完整。
- 后续做时间线审计、绩效分析、过程复盘时数据会有缺口。

### 6. 权限体系只在少数模块真正启用

- `dashboards` 有 `@RequirePermissions(...)`
- `settings` 的管理类接口有 `@RequirePermissions("settings.manage")`
- 但 `customers`、`research`、`website-analysis`、`scoring`、`commercial`、`emails`、`knowledge` 等主要业务接口基本没有权限装饰器

结论：

- 现在更多是“登录 + 数据域隔离”，不是“完整的细粒度权限控制”。

### 7. 智能识别结果没有完整回写到客户主档

- 官网分析会生成 `detectedLanguage`、`pricePositioning` 等分析结果
- 但客户主档 `country/language/timezone/currency` 主要仍靠人工录入/编辑

结论：

- “录入客户官网后，系统自动识别并回填客户基础信息”的体验还没完全落地。

### 8. 邮件归档只是基础版，不是完整线程中心

- 发送邮件时会新建 `EmailThread`
- 再次发送 follow-up 时没有复用旧线程
- IMAP 收件同步仅按 `inReplyTo` 或联系人邮箱去找最近线程
- 附件没有完整同步落库

结论：

- “往来自动归档”有基础能力，但线程连续性和附件完整度还不够。

### 9. AI 与公开搜索依赖环境配置，未配置时会降级

- AI 没有 `OPENAI_API_KEY` 时，`AiProviderService` 会返回占位内容
- 公开搜索没配 `SEARCH_PROVIDER`/`SEARCH_API_KEY` 时，研究报告只用官网和 CRM 数据

结论：

- 代码链路存在，但环境没配齐时，背调/评分/邮件质量会明显下降。

- 页面分区：公司信息、品牌资料、产品资料、OEM 能力、资质证书、成功案例、邮件素材

后端链路：

- 控制器：`apps/api/src/modules/knowledge/knowledge.controller.ts`
- 服务：`apps/api/src/modules/knowledge/knowledge.service.ts`
- 数据模型：`CompanyProfile`、`Brand`、`Product`、`OemCapability`、`Certificate`、`CaseStudy`、`EmailMaterial`

状态：`已完成`

已做完：

- 企业公司资料的查看/维护
- 品牌、产品、OEM 能力、证书、案例、邮件素材的创建、列表、更新
- 这些资料已被背调、评分、AI 邮件上下文引用，不是孤立数据

需求对应：

- 统一收纳目标客户数据
- 划分客户权限、开发状态
- 智能识别客户国家、语言、时区、货币、认证要求等

前端链路：

- `apps/web/src/pages/CustomersPage.tsx`
- `apps/web/src/pages/CustomerDetailPage.tsx`

后端链路：

- 控制器：`apps/api/src/modules/customers/customers.controller.ts`
- 服务：`apps/api/src/modules/customers/customers.service.ts`
- 数据模型：`Customer`、`Contact`、`CustomerStageHistory`、`CustomerAssignmentHistory`

状态：`部分完成`

已做完：

- 客户列表、客户新建、客户详情查看、客户资料编辑
- 客户来源、客户类型、负责人、阶段、标签、备注等基础字段管理
- 联系人新增
- 阶段手工修改
- 基于 `dataScope` 的客户数据隔离

做了一半：

- 客户字段虽然已经覆盖名称、官网、国家、语言、时区、币种、来源、类型、负责人、标签、备注，但还没有完整覆盖需求文档里提到的地址、社媒、强制认证要求等信息

- 客户合并/去重治理

- 强制认证要求自动判定

结论：

- “客户基础管理”已经完成。
- “智能识别规则”和“导入治理”还没完成。

## 4.3 客户背景调查模块

需求对应：

- 自动生成标准化背调报告
- 固定结构：公司信息、背景、产品线、竞争格局、品牌策略、价格定位、官网产品分析、总结建议

前端链路：

- 客户详情页 `research` 标签：`apps/web/src/pages/CustomerDetailPage.tsx`

后端链路：

- 控制器：`apps/api/src/modules/research/research.controller.ts`
- 服务：`apps/api/src/modules/research/research.service.ts`
- Processor：`apps/api/src/modules/research/research.processor.ts`
- 搜索服务：`apps/api/src/modules/research/search-provider.service.ts`
- 数据模型：`ResearchReport`、`AiGenerationRun`、`AiContentVersion`

状态：`部分完成`

已做完：

- 可以从客户详情触发背调任务
- 后端会组合客户信息、官网分析结果、企业资料库、联系人、历史邮件、公开搜索结果
- AI Prompt 已明确约束输出固定结构 JSON + Markdown 报告
- 报告会落库，并可在前端查看
- 报告保留 AI 版本记录

做了一半：

- 报告结构是有的，但很多字段仍依赖 LLM 总结，不是严格字段级采集
- `publicSearch` 依赖外部搜索配置，未配置时报告范围会降级
- 报告可读性与质量依赖 AI 和搜索结果，不是完全稳定的规则引擎
- 需求里的很多细项，例如发展节点、品牌迭代、市场覆盖、竞争品牌、营销话术等，不是独立结构化字段沉淀，而更像报告文本内容

没做：

- 背调结果人工校正后的结构化回写机制
- 背调模板管理
- 报告导出能力

结论：

- “能生成标准化背调报告”已经具备。
- “报告字段级结构化沉淀 + 稳定识别”还没有做透。

## 六、建议优先修复顺序

建议先修下面 6 件事，这 6 件最影响“系统能否真正上线使用”：

1. 跟进页断链修复

3. 自动跟进规则补齐

## 七、最终判断

这套系统不是“还没开始做”，而是已经完成了相当多的核心代码，尤其是一期基础架构和二三期的智能能力骨架都已经搭起来了。

但如果严格对照需求文档里的“客户背调 -> 官网分析 -> OEM 匹配 -> AI 邮件 -> 邮件往来 -> 跟进提醒 -> 报价打样 -> 管理看板”的完整闭环，当前更准确的状态是：

- 基础架构：`已完成`
- 智能分析：`部分完成`
- 邮件体系：`部分完成`
- 跟进与转化闭环：`部分完成`
- 安全合规与权限治理：`部分完成`

也就是说，这个项目目前最像“一个已经能跑起来、也有不少真功能的 MVP/一期系统”，但离需求文档里描述的“可稳定上线、可长期运营、闭环完整”的最终版，还有一段明确的收尾工作要做。
